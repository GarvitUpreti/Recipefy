3. ABOUT PROJECT
================================================================================


3.0.1 Project Overview and Objectives

Recipiefy is a client-side single-page application (SPA) developed for discovering and exploring meals, recipes, and related metadata through a structured browsing and search experience. In everyday use, people increasingly expect fast, browser-based tools that work across devices without installing native apps. Traditional approaches (static pages with no integration, or ad hoc searching across many sites) lack a focused, consistent user experience. This project addresses that gap for the recipe domain by combining React-based UI, declarative routing, and integration with TheMealDB—a public REST API providing meal entities, categories, areas, images, ingredients, and instructions.

Recipiefy was conceived and implemented as the main practical exercise for learning front-end engineering: component design, asynchronous data access with REST APIs, client-side routing, responsive layout, and basic quality habits (version control, structure, and honest scoping). If you are also submitting this for an internship or college, you can add one sentence naming the organisation or course here.

Project objectives:
●	Data discovery and browsing: Allow users to explore meals by category, by geographical/cuisine area, via text search, and via random discovery, with clear navigation between list and detail views.
●	Reliable integration with remote data: Centralise HTTP access in a dedicated service layer, handle loading and empty states, and avoid brittle coupling of fetch logic to every UI file.
●	Structured information architecture: Use React Router so URLs reflect user location in the app (home, categories, areas, search, meal details), supporting shareable paths and predictable back/forward behaviour.
●	Responsive, modern UI: Deliver a coherent layout using Tailwind CSS utilities, readable typography for long recipe text, and layouts that adapt to common mobile and desktop widths.
●	Next learning steps: Add **unit and component tests** (Jest / React Testing Library) with mocked API calls; keep extending manual regression checks when features change. Production deployment on Vercel is already in place for the current build (see §3.4).


3.0.2 System Architecture and Design

The architecture follows modern front-end practice: a React SPA compiled and bundled by Create React App, with separation between route-level pages, reusable components, API services, and shared hooks.

Logical layers (client-side):

●	Presentation layer: React function components and JSX; responsible for rendering UI from state/props and user events.
●	Navigation layer: react-router-dom routes, links, URL parameters (e.g. meal id), and programmatic navigation where needed.
●	Data access layer: Service module(s) encapsulating TheMealDB base URL, endpoint paths, query parameters, and normalised return values (arrays or single meal objects).
●	State and effects layer: React hooks (useState, useEffect, and custom hooks such as a reusable fetch helper) coordinating when to request data and how to reflect loading/error/empty outcomes.

Design principles:
●	Components are composed from smaller units (e.g. meal cards, lists, loaders, header/footer) rather than monolithic screens.
●	Pages orchestrate route parameters and call services; presentation components stay as “dumb” as practical.
●	The API contract (JSON field names from TheMealDB) is handled primarily in services and mapping utilities, reducing duplication.

3.0.3 Feature Implementation and Technical Details

Core user journeys implemented (some may still be refined as you keep learning):

●	Home and global navigation: Entry to the app with consistent header/footer and navigation to major sections.
●	Browse by category: List meal categories; filter meals by selected category; open meal detail.
●	Browse by area: Explore cuisine areas; list meals for an area; open meal detail.
●	Search: Search meals by name (and related search patterns supported by the service layer, such as first-letter browsing if exposed in UI).
●	Meal detail: Display title, imagery, ingredients with measures where available, and step-by-step instructions.
●	Random meal: Fetch a random recipe for exploration; ensure UI handles repeated requests and loading states.

Technical implementation highlights:
●	Asynchronous flows use async/await with the Fetch API; try/catch or equivalent guarding returns safe defaults (empty arrays / null) to avoid crashes on failed network calls.
●	List views map over meal summaries; detail views resolve full meal records by id when required.
●	Shared UI elements (loaders, empty states) reduce duplicate conditional rendering across screens.
●	Optional helpers (e.g. ingredient extraction from strIngredientN / strMeasureN fields) keep detail views readable.


3.0.4 Technology Implementation

Front-end stack (summary):
●	React 18: Component model, concurrent features available in the runtime, hooks-first development.
●	JavaScript (ES modules): Application source, JSX, modern syntax (async/await, destructuring).
●	react-router-dom v6: Client-side routing for SPAs.
●	Tailwind CSS + PostCSS + Autoprefixer: Utility-first styling and cross-browser CSS output.
●	Create React App (react-scripts): Development server, ESLint presets, production minification.
●	Fetch API: REST calls to TheMealDB JSON endpoints.
●	npm / Node.js: Dependency and script management; Git for version control.

Backend note: There is no custom Flask/Django server in this learning project; data comes from TheMealDB. Future enhancements (user accounts, saved favourites in your own database) would mean learning backend basics separately—outside the current scope.


3.0.5 Security and Performance Considerations

Security (front-end scope):
●	No password storage or server sessions in this project; sensitive credentials are not applicable unless a backend is added later.
●	API content is loaded over HTTPS when calling TheMealDB’s HTTPS endpoints from the browser.
●	If instructional text from the API were ever injected as raw HTML, sanitisation would be required to mitigate XSS; prefer rendering as plain text unless sanitisation is guaranteed.
●	Dependency hygiene: periodic review of npm audit advisories; upgrade paths documented in README over time.

Performance:
●	Avoid redundant refetching where possible (correct dependency arrays in hooks).
●	Use loading indicators so perceived latency is manageable on slower networks.
●	Image loading: rely on API-provided thumbnails; optional future work includes lazy loading below the fold.
●	Production build (`npm run build`) produces optimised bundles; measuring bundle size is an optional later lesson once the app runs well in development.

3.3 TESTING
================================================================================

3.3.1 Testing Methodology and Approach

The testing strategy for **Recipiefy** was designed to give **practical quality assurance** appropriate to a **learning project**: strong emphasis on **functional correctness** and **user-visible behaviour** on real browsers, while staying honest that **automated unit and component tests are not yet implemented** (planned as the next learning step). Because Recipiefy is a **client-only React SPA** that reads data from **TheMealDB** over HTTPS—without a custom backend, database, or user authentication in this scope—the testing narrative differs from a full-stack expense tracker: there are **no server-side forms, sessions, or SQL queries to validate** in this codebase; instead, testing focuses on **routing, API integration, rendering, loading/empty states, and the production deployment on Vercel**.

Testing philosophy applied to this project:
●	**Early and continuous manual checks:** Issues were caught while building each screen (navigation, lists, detail, search) rather than only at the end.
●	**Layer-appropriate coverage:** The “layers” here are **UI → router → service/fetch → JSON handling → render**; validation targets those boundaries.
●	**User-centric scenarios:** Tests follow real tasks a user would perform (find a category, open a meal, search, try random meal).
●	**Manual now, automation next:** Manual testing is **documented as completed** for the agreed scenarios; **Jest + React Testing Library** (included with Create React App) is the intended path for **unit/component tests** with **mocked `fetch`** so tests do not depend on live API availability.
●	**Environment parity:** Behaviour was checked on **local development** (`npm start`) and on the **deployed** build at **https://recipefy-nine.vercel.app/** so that production routing and assets behave as expected for typical browsing (including refresh on nested routes, which is a common SPA pitfall).

Testing environment setup:
●	**Local:** React development server, modern browser (Chrome/Firefox/Edge), **DevTools** (Console, Network, Performance) for debugging fetch failures, slow networks, and render issues.
●	**Production-like:** **Vercel** hosts the static output of `npm run build`; smoke tests confirm the live site loads, navigates, and fetches meal data over HTTPS.
●	**External dependency:** TheMealDB is a **public network dependency**; tests that would run in CI should **not** require it to be up 100% of the time—hence the plan to **mock** API calls in future automated tests.


3.3.2 Functional Testing Implementation

Functional testing for Recipiefy focused on **end-to-end user flows** from the home experience through category/area browsing, search, meal details, and random discovery. The following areas were exercised **manually**; results are summarised again in §3.3.5.

**Global navigation and layout**
Testing confirmed that the **header/navigation** links reach the expected sections, the **footer** and global layout remain consistent, and the app does not throw unhandled errors on first load. **Internal navigation** (React Router) was checked so that moving between pages updates the view without a full page reload, and the user can understand **where they are** in the app.

**Browse by category**
Testing verified **loading indicators** while categories load, successful rendering of category grids/lists, navigation into a **filtered meal list** for a chosen category, and navigation into **meal detail**. **Empty or sparse results** were observed where applicable so the UI does not assume meals always exist.

**Browse by area (cuisine / region)**
Parallel checks were performed for **area listing**, **meals for an area**, and **meal detail**, ensuring URLs and route parameters behave consistently with category flows.

**Search behaviour**
Search testing covered **successful queries** returning meals, **queries with no matches** so an appropriate **empty state** is shown, and basic **input edge cases** (e.g. empty search, repeated searches) so the experience remains stable. Exact search rules follow TheMealDB’s API behaviour; the UI’s job is to **not crash** and to **communicate** when nothing is found.

**Meal detail view**
Detail testing focused on **title, image, ingredients and measures, and instructions** being readable, on long text **wrapping** sensibly, and on the page remaining usable on **smaller viewports** (responsive check in the browser).

**Random meal**
Random meal was tested for **repeated use** (new request / new result flow) and for **loading** so the user does not see stale content without feedback.

**Consistency between local and deployed builds**
Because deployment introduces **routing and caching** differences, the same core scenarios were re-checked on **https://recipefy-nine.vercel.app/** after deployment.


3.3.3 Security Testing Implementation

Recipiefy does **not** implement proprietary login, cookies for auth, or server-side templates—so classical **session fixation**, **CSRF on forms**, or **SQL injection against our database** are **not applicable** in this project’s current architecture. Security-related verification instead emphasises **realistic front-end risks** and **supply-chain hygiene**:

●	**Transport security:** The live app on **Vercel** is served over **HTTPS**; API calls to **TheMealDB** use HTTPS URLs from the service layer.
●	**Cross-site scripting (XSS) awareness:** Meal instructions and descriptions originate from **third-party JSON**. If content were ever injected into the DOM as raw HTML, that could introduce XSS; **safe rendering practices** (plain text / structured React nodes, or sanitised HTML only if explicitly chosen) are part of ongoing vigilance—not server-side “HTML escaping” as in server-rendered forms, but **equally important** on SPAs when rendering rich text.
●	**Dependency and supply chain:** JavaScript dependencies are managed via **npm**; periodically running **`npm audit`** (and upgrading responsibly) reduces exposure to known vulnerable packages.
●	**Secrets:** No API keys are required for TheMealDB’s public endpoints as used here; **no secrets should be committed** to the repository—good discipline for future features that might require keys.
●	**Browser security context:** Understanding **CORS** helps interpret failures when experimenting with APIs from new origins; the current project uses a public API designed for browser use in typical configurations.

This section is **proportionate** to a learning SPA. A full **penetration test** or **formal security audit** is out of scope; the project instead shows **awareness** of the main client-side concerns.


3.3.4 Performance and Usability Testing

**Performance (informal, developer-led)**  
Performance validation relied on **browser tooling** rather than formal load-testing infrastructure:
●	**Network throttling** (e.g. Slow 3G in DevTools) to confirm **loaders** appear and the UI remains responsive enough during waits for TheMealDB.
●	**Repeated navigation** between list and detail screens to spot unnecessary **refetch behaviour** or UI jank tied to hook dependencies.
●	Awareness that **`npm run build`** produces **optimised bundles** for production; deeper metrics (bundle analysers, Lighthouse scores logged over time) remain optional enhancements.

**Usability**
Usability checks emphasised **clarity of navigation**, **readability** of recipe content, **tap targets** on smaller screens where Tailwind breakpoints apply, and **consistent feedback** when data is loading or absent. Full WCAG conformance auditing was **not claimed**; iterative accessibility improvements (semantic headings, contrast, keyboard focus) remain good next steps.

3.4 DEPLOYMENT
================================================================================

3.4.1 Deployment Strategy and Planning

The deployment strategy for Recipiefy aimed at a **simple, reliable pipeline** suited to a **Create React App** production build: compile and bundle the SPA, serve static assets over **HTTPS**, and support **client-side routing** on refresh. **Vercel** was selected as the hosting platform for its **tight Git integration**, **generous free tier for learning projects**, **automatic HTTPS**, and **first-class support for static front-end deployments**. Unlike a Flask application backed by PostgreSQL on Render (as in unrelated full-stack examples), Recipiefy’s backend is **TheMealDB over HTTPS**, so deployment centres on **build correctness**, **routing configuration**, and **post-deploy smoke testing**.

Deployment objectives included:
●	A stable **public URL** for demos and portfolios.
●	Repeatable deploys when updating the Git repository.
●	Confidence that **`npm run build`** succeeds before promoting changes.


3.4.2 Production Environment Configuration

**Application build configuration**  
●	Source code is bundled with **`npm run build`** via **react-scripts**, producing optimised JS/CSS in the **`build/`** directory.
●	Runtime configuration is intentionally minimal because secrets are not embedded for public API usage as implemented.

**Hosting configuration (Vercel)**  
●	The project is linked so that pushes (typically to **`main`** or the configured branch) trigger **install**, **`npm run build`**, and publication of assets.
●	**SPA routing:** Vercel should route unknown paths to **`index.html`** so refreshing on nested routes does not return raw **404** from the CDN—project settings should match CRA expectations if issues arise.

**External services**  
●	**TheMealDB** remains the authoritative meal dataset; Recipiefy does not deploy a private database alongside this learning scope.


3.4.3 Deployment Process Implementation

**Continuous deployment via Git**  
●	Pushing changes to the connected repository triggers Vercel’s **build pipeline**.  
●	This substitutes for manually uploading FTP artefacts—a practical introduction to **modern deployment workflows**.

**Build pipeline (conceptual)**  
●	Check out repository contents.  
●	Run **`npm install`** (or ci equivalent).  
●	Run **`npm run build`**.  
●	Publish **`build/`** output to Vercel’s CDN edge network.

**Smoke verification after deploy**  
●	Open **https://recipefy-nine.vercel.app/**  
●	Walk **home → browse flows → search → detail → random**.  
●	If routing breaks on refresh for detail URLs, verify **SPA fallback / rewrite** rules in Vercel.


3.4.4 Production Performance and Monitoring

Performance monitoring for this learning deployment is intentionally lightweight:
●	Use browser **DevTools Network** on production for occasional checks after releases.
●	Optional **Vercel Analytics** or similar may be enabled later; not required for baseline reports.

Because traffic is modest for a portfolio/demo app, **enterprise-grade SLAs** are neither measured nor claimed.

**Honest limits**
